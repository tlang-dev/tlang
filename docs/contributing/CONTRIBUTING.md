# Contributing to TLang

Thank you for your interest in contributing to TLang! This document provides guidelines for contributing to the project.

## Ways to Contribute

There are many ways to contribute to TLang:

- **Bug Reports**: Report bugs you encounter
- **Feature Requests**: Suggest new features or improvements
- **Documentation**: Improve existing docs or add new documentation
- **Code Contributions**: Submit pull requests with bug fixes or new features
- **Code Review**: Review open pull requests
- **Testing**: Help test new features and bug fixes
- **Examples**: Create example projects demonstrating TLang usage
- **Generators**: Create new language generators

## Getting Started

### Prerequisites

- Rust 1.70+ (recommended: latest stable)
- Cargo (comes with Rust)
- Git

### Building TLang

```bash
# Clone the repository
git clone https://github.com/joel-f/tlang.git
cd tlang

# Build in debug mode (for development)
cargo build

# Build in release mode (for production)
cargo build --release

# Run tests
cargo test

# Run clippy (code linter)
cargo clippy

# Run formatting check
cargo fmt --check
```

The compiled binary will be at `target/debug/tlang` (or `target/release/tlang` for release builds).

### Running TLang

You can run the compiled binary directly:

```bash
./target/debug/tlang --version
./target/debug/tlang --help
```

Or add it to your PATH:

```bash
export PATH="$(pwd)/target/debug:$PATH"
tlang --version
```

## Building the Website

The TLang website is itself generated by TLang — a dogfood proof-of-concept
using `doc [md, html]` templates. The generator lives in `website/`.

```bash
# Rebuild the runtime first if you changed any Rust source
cargo build

# Regenerate all website HTML and Markdown
./target/debug/tlang compile-run website/Main.tlang

# Output is written to website/docs/ (copied to docs/ by CI for GitHub Pages)
```

After regenerating, open `website/docs/index.html` in a browser to preview locally.
If you add new doc pages, wire them into `website/Main.tlang` and update the nav
in the `doc [html] htmlPage` template at the bottom of that file.

## Development Workflow

### Setting Up for Development

1. Fork the repository on GitHub
2. Clone your fork locally
3. Create a feature branch:
   ```bash
   git checkout -b feature/my-feature
   ```

### Making Changes

1. Make your changes in the appropriate files
2. Ensure your code follows the project's style (run `cargo fmt`)
3. Add tests for new functionality
4. Ensure all existing tests pass
5. Run clippy to catch potential issues:
   ```bash
   cargo clippy -- -D warnings
   ```

### Committing Changes

Follow these guidelines for commit messages:

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Fix bug" not "Fixes bug")
- Limit the first line to 72 characters or less
- Reference issues and pull requests liberally after the first line
- Consider starting the commit message with an applicable emoji:
  - `✨` for new features
  - `🐛` for bug fixes
  - `📝` for documentation changes
  - `🔧` for configuration changes
  - `⚡` for performance improvements
  - `🎨` for UI/UX changes
  - `🚧` for work in progress
  - `✅` for tests
  - `📦` for dependency updates

Example:
```
✨ Add support for for-each loops in templates

Implements the ability to iterate over lists in template blocks
using the <[ ... ]> syntax.

Fixes #123
```

### Submitting a Pull Request

1. Push your changes to your fork:
   ```bash
   git push origin feature/my-feature
   ```

2. Open a pull request on GitHub:
   - Go to the [TLang repository](https://github.com/joel-f/tlang)
   - Click "New pull request"
   - Select your fork and branch
   - Add a clear title and description
   - Reference any related issues

3. Ensure the PR passes all CI checks

4. Wait for code review and address any feedback

## Code Style Guidelines

### Rust Code

- Follow [Rust API Guidelines](https://rust-lang.github.io/api-guidelines/)
- Use `snake_case` for variables and functions
- Use `PascalCase` for types, traits, and enums
- Use `SCREAMING_SNAKE_CASE` for constants
- Prefer immutability where possible
- Use `Result` for error handling (avoid `panic!` in library code)
- Document all public items with `///` doc comments
- Use `//` for single-line comments, `/* */` for multi-line

### TLang Code (in examples and tests)

- Use consistent indentation (4 spaces)
- Use `PascalCase` for template names
- Use `camelCase` for template parameters
- Use `snake_case` for helper functions
- Use `UPPER_CASE` for constants in model blocks
- Add comments explaining non-obvious logic

## Testing

### Running Tests

```bash
# Run all tests
cargo test

# Run specific test module
cargo test <module_name>

# Run with verbose output
cargo test -- --nocapture
```

### Adding Tests

- Add new tests in the appropriate test module
- Use descriptive test names
- Test both happy paths and error cases
- Keep tests focused on one thing each

### Conformity Tests

TLang uses conformity tests to ensure the language behaves as specified. These are located in `src/conformity_tests.rs`.

Each conformity test:
1. Defines a TLang program as a string
2. Compiles it
3. Runs it
4. Checks the output

Add new conformity tests when:
- Adding new language features
- Fixing bugs in existing features
- Clarifying edge cases in the specification

### Writing Documentation

- Use Markdown format
- Keep documentation up-to-date with code changes
- Use code examples liberally
- Document edge cases and common pitfalls
- Link to related documentation

### Documentation Structure

- **README.md**: High-level project overview
- **docs/specs/**: Language specifications

### Updating Documentation

When making code changes, check if any documentation needs to be updated:

1. If you add a new feature, add documentation for it
2. If you change existing behavior, update the relevant docs
3. If you fix a bug, check if the bug was documented and update accordingly

## Creating Generators

Generators are TLang packages that know how to render template output to a specific target language or format.

### Generator Structure

A generator is a TLang project with:

1. A `manifest.yml` with a `main:` entry point
2. A main file that exposes the `generate` function
3. Logic to transform TLang template instances into target language code

### Example Generator (Kotlin)

```yaml
# manifest.yml
name: Kotlin
project: KotlinGen
organisation: TLangGen
version: 1.0.0
stability: alpha
releaseNumber: 1
main: KotlinCodegen
```

```tlang
# KotlinCodegen.tlang
expose generate

helper {
    func generate(tmpl): String {
        // Implementation to render Kotlin code
        // from the template instance
    }
}
```

### Packaging a Generator

```bash
tlang package generators/kotlin
```

This compiles the generator, bundles it into a `.tbag` file, and publishes it to the local tbox.

## Reporting Issues

When reporting issues, please include:

1. **Description**: Clear description of the problem
2. **Steps to Reproduce**: How to trigger the issue
3. **Expected Behavior**: What you expected to happen
4. **Actual Behavior**: What actually happened
5. **Environment**:
   - Operating system
   - Rust version (`rustc --version`)
   - TLang version (`tlang --version`)
6. **Example Code**: Minimal code that reproduces the issue
7. **Error Messages**: Any error messages or stack traces

## Requesting Features

When requesting features, please include:

1. **Use Case**: What problem you're trying to solve
2. **Proposed Solution**: How you think it should be implemented
3. **Examples**: Code examples showing the desired syntax/behavior
4. **Alternatives**: Any alternative approaches you've considered
